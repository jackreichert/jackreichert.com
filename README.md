# jackreichert.com

Static build of jackreichert.com: content is written in WordPress (WordPress.com), synced to markdown, and built with Eleventy. The design is a revival of the 2020 jackreichert.com (see `DESIGN.md`; design intent in `PRODUCT.md`).

## Publishing flow (WP.com -> this repo)

- Publish/edit on WordPress.com.
- GitHub Actions polls WordPress.com every 15 minutes, runs `npm run sync`, and only commits/deploys when content actually changed.
- No WP plugin webhook is required for this flow.

## Commands

| Command | What it does |
|---|---|
| `npm run sync` | Pull posts, pages, and images from the WordPress API into `content/` and `assets/images/`. Images are resized (max 1600px), compressed, given 800w/1200w variants, and mirrored to WebP. Deterministic — git diff shows only real content changes. |
| `npm run optimize-images` | One-shot re-encode of existing `assets/images/` masters + width/WebP variants (useful after changing quality settings). |
| `npm run dev` | Eleventy dev server on :8087 (via `.claude/launch.json`) or default port. Search is inert in dev (no Pagefind index) and the WP stats beacon does not fire on localhost. |
| `npm run build` | Production build: Eleventy (minified+hashed CSS, responsive `<picture>`/srcset) → `_site/`, then Pagefind indexes it. |

### Performance pipeline

- **Sync / optimize-images**: resize + compress masters, emit `*-800w.*` / `*-1200w.*`, emit `.webp` companions.
- **Build**: minify + content-hash CSS (`style.<hash>.css`), wrap images in WebP `<picture>`, lazy-load body images, preload LCP WebP + critical fonts only.
- **Deploy**: every push to `main` builds and deploys; scheduled runs sync WP and deploy only when content changed.
- **WP stats**: still loaded, but only after `window.load` + `requestIdleCallback` (never on localhost).
- **Gzip/Brotli**: left to GitHub Pages / Cloudflare at the edge — not prebuilt in the repo.

## Architecture

- **Content**: `content/posts/*.md`, `content/pages/*.md` — generated by `scripts/sync.mjs`, never edited by hand (the sync wipes and rewrites them).
- **Identity/config**: `_data/site.js` — the single source for site name, tagline, author, canonical URL, WordPress source site, and stats blog ID. Loaded by Eleventy as the `site` global and imported by the sync script.
- **Design**: `assets/css/style.css` + `_includes/*.njk` + the root templates (`index.njk`, `archive.njk`, `categories.njk`, `search.njk`, `404.njk`). Documented in `DESIGN.md`.
- **Stats**: the WordPress.com stats beacon in `_includes/base.njk` reports views (per-post, via `wp_id` frontmatter) to the WP stats dashboard. It's disabled on localhost, and entirely off when `site.wpcom.blogId` is null.
- **Automation**: `.github/workflows/sync-and-deploy.yml` polls for WP changes on a 15-minute schedule, syncs content, and deploys only when sync produced a diff.

## Standing up another site on this pattern

1. Copy the repo (minus `content/`, `assets/images/`, `_site/`).
2. Edit **`_data/site.js`**: name, tagline, author, url, `wpcom.site` (the WordPress.com domain), and `wpcom.blogId` (`https://public-api.wordpress.com/rest/v1.1/sites/<domain>` → `"ID"`; null to disable stats).
3. Run `npm install && npm run sync` — content and images arrive from the new WP site, each post carrying its `wp_id`.
4. Replace the design: `assets/css/style.css` and the `_includes/` + root templates. Templates read all identity from `site.*` and all content from the synced frontmatter (`title`, `date`, `permalink`, `wp_id`, `categories`, `post_tags`, `description`, `featured_image`, optional `featured_caption`), so no template carries site-specific strings.
5. `npm run build` and deploy `_site/`.
